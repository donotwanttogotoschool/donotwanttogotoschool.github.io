最近在做一个智能客服项目，越往后做越觉得，Agent 的评估不能只看“最后这句话答得对不对”。
因为它不像普通 LLM 那样只是输入文本、输出文本，它中间还会自己做步骤规划、调用工具、查外部系统，所以如果只看最终回答，很容易把一些问题漏掉。

目前我自己的理解是，评估 Agent 至少要看这几件事：

  • 任务最后到底有没有完成
  • 工具是不是调对了
  • 中间流程有没有跑偏
  • 成本和耗时能不能接受
  • 如果失败了，问题出在哪一步

下面是我查资料以后整理出来的一些点，后面做项目评估的时候应该能直接用上。

1. 为什么 Agent 需要单独评估

传统的 LLM 评估，很多时候关注的是“回答质量”，比如答得准不准、像不像人写的、有没有事实错误。
但 Agent 不一样，它是一个“会行动”的系统，所以评估对象也不能只停留在最后一句回复，而是要扩大到整个行为过程。

我看到的资料里提到，Agent 的评估通常要覆盖这些维度：

  • 任务成功率
  • 是否理解用户真实意图
  • 工具调用是否正确
  • 安全与合规
  • 是否存在偏见
  • 成本和效率表现

也就是说，Agent 评估更像是在看“这个系统靠不靠谱”，而不只是“这句话写得好不好”。(Amazon Web Services, Inc.)

2. Agent 评估的一般流程

目前比较常见的一套流程，大概可以分成四步：

  1. 先定义评估目标和指标
  2. 准备测试数据，最好来自真实业务场景
  3. 执行评估并分析结果，可以结合人工评估和 LLM-as-Judge
  4. 根据结果继续补测试集，反复迭代

这里我觉得最关键的一点是：
如果任务本身是多步骤的，那最好把完整执行轨迹也记录下来。这样不只是看最终回答对不对，还能顺便检查：

  • 工具有没有选对
  • 参数有没有传对
  • 上下文有没有带完整
  • 最终结论是不是和事实一致

这个对客服类 Agent 尤其重要，因为它经常会查数据库、调接口、走规则。(Amazon Web Services, Inc.)

3. 常见指标体系

我把资料里的指标简单分成三类：

业务指标：任务完成率、决策准确率、工具调用正确率
效率指标：平均任务耗时、平均交互轮数
安全与伦理指标：偏见发生率、合规风险等

如果从工程落地的角度看，我觉得最值得优先关注的是这两个：

  • Tool Call Accuracy 看工具有没有选对，参数提取得对不对。
  • Task Completion Rate 看最终业务目标有没有真的完成。

前者更像是在看 Agent “会不会做事”，后者是在看它“有没有把事做成”。这两个指标如果都不稳定，后面用户体验大概率也不会太好。(Amazon Web Services, Inc.)

4. 常见的 Agent 评估框架

我看到几种比较常被提到的框架：

框架 我理解的重点
AgentBoard 更适合看过程，能分析轨迹、进度率、Grounding Accuracy 等
AgentBench 更适合做横向比较，覆盖 OS、数据库、知识图谱、网页浏览等多种环境
τ-bench 更偏真实业务对话，适合客服、零售、航旅这类场景
GAIA / WebArena 更偏通用复杂任务，强调多模态和 Web 操作能力

按我的理解：

  • 如果想看“它到底哪一步错了”，可以偏向 AgentBoard
  • 如果想看“不同模型或 Agent 谁更强”，可以看 AgentBench
  • 如果是像我这种在做客服方向的项目，τ-bench 会更贴近真实场景

所以后面如果真要做系统性的评估，我大概率会优先关注更接近真实业务流程的方案。(Amazon Web Services, Inc.)

如果后面要把这部分写成博客，或者做项目汇报、答辩，我感觉下面这张表会很有用。
因为它不只是列论文名，而是能顺手说明“这个 benchmark 到底适合拿来讲什么”。

论文 / Benchmark 顶会 适合博客引用的点
τ-bench: A Benchmark for Tool-Agent-User Interaction in Real-World Domains ICLR 2025 真实业务域里的“用户-Agent-工具-规则”交互；用最终数据库状态评估任务完成，提出 pass^k 看稳定性。(arXiv)
AgentBench: Evaluating LLMs as Agents ICLR 2024 早期经典 Agent 综合评测，覆盖 OS、数据库、知识图谱、网页购物、网页浏览等交互环境。(ICLR Proceedings)
AgentBoard: An Analytical Evaluation Board of Multi-turn LLM Agents NeurIPS 2024 Datasets & Benchmarks 多轮 Agent 轨迹分析，强调 progress rate、grounding、可视化诊断，不只看最终成功率。(NeurIPS 会议论文集)
WebArena: A Realistic Web Environment for Building Autonomous Agents ICLR 2024 真实网页环境中的长程任务，适合说明“真实业务操作比静态 QA 难很多”。(ICLR Proceedings)
VisualWebArena: Evaluating Multimodal Agents on Realistic Visual Web Tasks ACL 2024 WebArena 的多模态扩展，适合讲视觉网页、截图理解、UI 操作类 Agent 评估。(ACL 2024)
GAIA: A Benchmark for General AI Assistants ICLR 2024 评估推理、多模态、网页浏览、工具使用等通用助手能力；人类与 GPT-4 插件版差距很大。(ICLR Proceedings)
Mind2Web: Towards a Generalist Agent for the Web NeurIPS 2023 Datasets & Benchmarks 通用网页 Agent 数据集，任务来自真实网站，适合写“网页型业务 Agent”的评估。(NeurIPS 会议论文集)
ToolLLM / ToolBench: Facilitating LLMs to Master 16000+ Real-world APIs ICLR 2024 工具调用训练与评估，覆盖大量真实 API，适合讲 Tool Call Accuracy / Tool Use。(ICLR Proceedings)
API-Bank: A Comprehensive Benchmark for Tool-Augmented LLMs EMNLP 2023 工具增强 LLM 的规划、检索、调用能力评估，包含可运行 API 工具系统。(ACL Anthology)
BizBench: A Quantitative Reasoning Benchmark for Business and Finance ACL 2024 最贴近“业务 / 金融推理”的顶会 benchmark，评估财务数据上的数量推理和程序合成。(ACL Anthology)
SWE-bench: Can Language Models Resolve Real-world Github Issues? ICLR 2024 Oral 虽然偏软件工程，但很适合类比企业任务：从真实 GitHub issue 到 patch，再用测试验证最终状态。(OpenReview)

如果只从我现在这个客服 Agent 项目的角度看，我会优先关注这几类：

  • τ-bench：因为它最接近真实业务流程，会看用户、Agent、工具和规则之间的交互。
  • AgentBoard：适合分析过程错误，尤其是想查清楚“到底哪一步出了问题”。
  • ToolLLM / ToolBench 和 API-Bank：比较适合补“工具调用能力”这块视角。
  • SWE-bench：虽然不是客服场景，但它那个“看最终状态是否真的被修好”的思路很值得借鉴。

6. 顺手记一下 OpenRouter 是什么

顺着评估这个话题，我觉得还可以补一句 OpenRouter。
如果后面我要测试不同模型在同一套 Agent 流程里的表现，那像 OpenRouter 这种统一模型接入层就会比较方便。

按我现在的理解，OpenRouter 更像是一个“模型路由和聚合入口”：

  • 它方便把不同来源的模型接到同一套调用流程里
  • 做实验的时候更容易切换模型，不用反复改太多业务代码
  • 如果想做多模型对比，它能降低接入成本

对学生项目来说，它最大的价值不是“直接帮你完成评测”，而是让你更容易做这些事：

  • 在同一份测试集上比较不同模型的效果
  • 观察不同模型在工具调用、成本、延迟上的差异
  • 给同一个 Agent 工作流做 A/B 测试

不过这里也要分清楚一件事：
OpenRouter 本身不是 benchmark，也不是评估框架。
它更像是实验平台里的“模型接入层”，真正的评估标准还是得靠你自己定义指标、整理数据集、记录轨迹和分析结果。

如果后面要把这部分写进博客，我会更愿意把 OpenRouter 介绍成：

一个方便做多模型实验和对比的工程工具，而不是评价 Agent 好坏的标准本身。

7. 对项目落地比较有用的做法

资料里给的建议，我觉得对实际开发挺有参考价值：

  • 优先从真实业务数据里整理测试集
  • 如果暂时没有真实数据，可以先人工构造种子样例
  • 之后再用 self-instruct 之类的方法扩测试集
  • 评估时同时看“工具调用细节”和“最终状态是否正确”
  • 对失败样本做归因分析，而不是只记一个失败结果

这一点对我启发挺大。
以前可能会习惯只看最后有没有答对，但现在感觉更应该把失败样本拆开看：到底是理解意图错了、工具选错了、参数错了，还是规则没遵守。这样后面改系统的时候才知道该优化哪一层。(Amazon Web Services, Inc.)

8. 两个我觉得比较有代表性的案例

案例一：真实业务流程方向

有一个案例是基于 τ-bench + Strands Agents + Langfuse 去复现零售客服 Agent，重点评估：

  • 数据一致性
  • 失败归因
  • 任务耗时
  • Token 消耗

这个就比较像真实工程场景，不只是看最终回复，而是把整个任务过程都纳入评估。(Amazon Web Services, Inc.)

案例二：过程分析方向

另一个案例是基于 AgentBoard 做天气报告助手评估，使用的指标包括：

  • Success Rate
  • Progress Rate
  • Grounding Accuracy
  • Score State
  • 难度分层

这个案例让我感觉比较适合做“过程诊断”。不过资料里也提到，AgentBoard 的 Grounding Accuracy 只能说明动作没有报错，不一定代表工具选择就是最优的，所以看指标的时候还是不能太机械。(Amazon Web Services, Inc.)

如果后面整理成博客标题

我暂时想到一个还比较合适的标题：

从“回答正确”到“行为可靠”：Agent 质量评估体系与工程实践

我现在的一句话理解

Agent 评估的重点，不是单纯问“模型答得好不好”，而是看它能不能在正确的上下文里，调用正确的工具，遵守正确的规则，并且以一个还能接受的成本，把任务稳定做完。
