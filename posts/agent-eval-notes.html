Agent 评估整理：从 benchmark 到 OpenRouter

最近在做智能客服项目，发现 Agent 评估真的不能只看最后一句答得对不对。这篇笔记把我查到的评测框架、常见 benchmark 和工程工具理解，按项目视角整理了一遍。

日期
2026.05.04

项目
智能客服 / Agent 评估

标签
Benchmark / Tool Use / OpenRouter

我现在的核心理解

Agent 的评估重点，不只是问“模型答得好不好”，而是看它能不能在正确的上下文里，调用正确的工具，遵守正确的规则，并且以还能接受的成本，把任务稳定做完。

所以对我这种正在做客服项目的人来说，最后一句回复对不对只是结果，真正要看的还包括中间过程有没有跑偏、工具有没有调错、失败到底卡在哪一步。

为什么 Agent 需要单独评估

普通 LLM 评估很多时候关注的是回答质量，但 Agent 会自己做规划、调工具、查外部系统，所以只看文本输出很容易把问题漏掉。更合理的做法，是把它当成一个会执行任务的系统来看。

  • 任务最后有没有完成
  • 工具是不是调对了
  • 中间过程有没有偏离用户意图
  • 成本、延迟和稳定性是不是能接受
  • 失败能不能定位到具体原因

我整理的一张常见论文 / Benchmark 表

如果后面要把这部分写成博客或者答辩材料，这张表很好用。因为它不是只列论文名，而是顺手说明每个 benchmark 更适合拿来解释什么。

论文 / Benchmark 顶会 适合博客引用的点
τ-bench ICLR 2025 真实业务域里的“用户-Agent-工具-规则”交互；用最终数据库状态评估任务完成，并提出 pass^k 看稳定性。
AgentBench ICLR 2024 早期经典 Agent 综合评测，覆盖 OS、数据库、知识图谱、网页购物、网页浏览等交互环境。
AgentBoard NeurIPS 2024 Datasets & Benchmarks 多轮 Agent 轨迹分析，强调 progress rate、grounding、可视化诊断，不只看最终成功率。
WebArena ICLR 2024 真实网页环境中的长程任务，适合说明真实业务操作比静态 QA 难很多。
VisualWebArena ACL 2024 WebArena 的多模态扩展，适合讲视觉网页、截图理解、UI 操作类 Agent 评估。
GAIA ICLR 2024 评估推理、多模态、网页浏览、工具使用等通用助手能力，适合拿来说明“通用智能助理”还很难。
Mind2Web NeurIPS 2023 Datasets & Benchmarks 通用网页 Agent 数据集，任务来自真实网站，适合写网页型业务 Agent 的评估。
ToolLLM / ToolBench ICLR 2024 工具调用训练与评估，覆盖大量真实 API，适合补充 Tool Call Accuracy / Tool Use 这类视角。
API-Bank EMNLP 2023 工具增强 LLM 的规划、检索、调用能力评估，包含可运行 API 工具系统。
BizBench ACL 2024 更贴近业务 / 金融推理任务，评估财务数据上的数量推理和程序合成。
SWE-bench ICLR 2024 Oral 虽然偏软件工程，但很适合类比企业任务：从真实问题到 patch，再用测试验证最终状态。

如果只从我现在这个客服 Agent 项目的角度看，我会优先关注 τ-bench、AgentBoard、ToolLLM / ToolBench、API-Bank 这几类，因为它们要么更贴近真实流程，要么更适合分析工具调用问题。

项目落地时我更关心哪些指标

从论文里看，评估维度可以很多，但如果真的要在项目里跑起来，我觉得最该优先看的还是两个：

  • Tool Call Accuracy：工具有没有选对，参数提取得对不对。
  • Task Completion Rate：最终业务目标有没有真的完成。

前者更像是在看 Agent 会不会做事，后者是在看它有没有把事情做成。对于客服项目，这两个指标往往比单独看回答流畅度更关键。

OpenRouter 在这个问题里算什么

顺着评估这个话题，我也顺手记一下 OpenRouter。按我现在的理解，它更像是一个统一的模型接入层，而不是评估标准本身。

  • 方便把不同来源的模型接到同一套调用流程里
  • 做实验时更容易切换模型，不用来回改太多业务代码
  • 如果想做多模型对比，它能明显降低接入成本

对学生项目来说，它比较有价值的地方在于：

  • 可以在同一份测试集上比较不同模型效果
  • 可以观察不同模型在工具调用、成本、延迟上的差异
  • 可以给同一个 Agent 工作流做 A/B 测试

OpenRouter 不是 benchmark，也不是评估框架。它更像实验平台里的模型入口，真正的评估还是要靠你自己定义指标、整理测试集、记录轨迹和分析结果。

我现在觉得比较实用的做法

  • 优先从真实业务数据里整理测试集
  • 没有真实数据时，先人工构造种子样例
  • 同时看工具调用细节和最终状态是否正确
  • 失败样本要做归因分析，而不是只记“失败”

这也是我这次查资料后最大的变化：以前更容易只盯着最终回答，现在更倾向于把失败拆开看，分清楚是理解意图错了、工具选错了、参数错了，还是规则没遵守。
